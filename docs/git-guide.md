# План работы с Git: от простого к предприятию

## Уровень 1: Базовый (Simple)

### Что это такое?
Git — это система контроля версий. Представьте, что это "машина времени" для вашего кода: вы можете вернуться к любой версии файла в любой момент.

### Как начать?

```bash
# 1. Настройка (делается один раз)
git config --global user.name "Ваше Имя"
git config --global user.email "your@email.com"

# 2. Создание репозитория
git init my-project
cd my-project

# 3. Основной цикл работы
git add .                          # Добавить все изменения
git commit -m "Описание изменений" # Сохранить snapshot

# 4. Проверка состояния
git status    # Что изменилось?
git log       # История коммитов
```

### Простая команда для ежедневной работы:
```bash
git add . && git commit -m "фикс" && git push
```

## Уровень 2: Ветки (Branching)

### Что это такое?
Ветки — это параллельные вселенные. Можете работать над новой фичей, не ломая основную версию.

### Как использовать?

```bash
# Создать и переключиться на новую ветку
git checkout -b new-feature

# Работать как обычно
git add .
git commit -m "Добавил кнопку"

# Вернуться к главной ветке
git checkout main

# Смерджить (объединить) изменения
git merge new-feature
```

## Уровень 3: Удалённые репозитории (Remote)

### Что это такое?
Это ваш проект на GitHub/GitLab, где хранится код и работают командой.

### Команды:

```bash
# Подключить удалённый репозиторий
git remote add origin https://github.com/user/repo.git

# Загрузить свои коммиты
git push origin main

# Скачать чужие изменения
git pull origin main

# Синхронизировать локальные ветки с удалёнными
git fetch --all
```

## Уровень 4: Worktrees (Рабочие деревья)

### Что это такое?
**Worktrees** позволяет иметь несколько рабочих копий одного репозитория одновременно в разных папках.

**Проблема**: Вы работаете над фичей в `feature-A`, но нужно срочно починить баг в `main`.

**Решение**:
```bash
# 1. Создать новый worktree для ветки feature-B в соседней папке
git worktree add ../repo-feature-B feature-B

# 2. Перейти в него и работать
cd ../repo-feature-B

# 3. Удалить worktree (когда закончили)
git worktree remove ../repo-feature-B
```

## Уровень 5: Pull Requests (PR)

### Что это такое?
PR — это **запрос на слияние** вашего кода в основную ветку. Это этап проверки (code review) перед тем, как код попадет в `main`.

### Процесс:
1. Создали ветку.
2. Сделали изменения и заpushили.
3. Открыли PR в интерфейсе GitHub/GitLab.
4. Коллеги проверили, оставили комментарии.
5. Вы поправили (если надо) и нажали "Merge".

---

## Уровень 6: CI/CD (Continuous Integration/Deployment)

### Что это такое?
**CI/CD** — это автоматизация. Робот, который проверяет ваш код.

- **CI** (Integration) — автоматический запуск тестов и линтеров при каждом push.
- **CD** (Deployment) — автоматическая выкладка кода на сервер после успешных тестов.

---

## Уровень 7: Enterprise (Предприятие)

### Ключевые практики:

1. **Branch Protection**: Запрет пушить в `main` напрямую. Только через PR.
2. **Rebase**: Использование `git rebase` для чистой истории коммитов.
3. **Git Hooks**: Скрипты, которые проверяют код локально перед коммитом (например, `Husky`).
4. **Code Ownership**: Назначение ответственных за определенные части кода.
